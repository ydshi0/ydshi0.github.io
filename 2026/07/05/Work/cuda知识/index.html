一. GPU 架构

1.1 SM（Streaming Multiprocessor）

GPU 的核心计算单元，一个 GPU 含有多个 SM。每个 SM 包含：

  • 共享内存（Shared Memory）：SM 内线程块可共享的高速内存
  • 寄存器（Registers）：每个线程的私有存储
  • 调度器（Warp Scheduler）：管理线程束的执行

1.2 线程层级

CUDA 将并行计算组织为三级逻辑结构：Thread、Block、Grid

Block 内线程可通过 shared memory 通信，每个Block最多包含1024个Thread

层级与硬件的映射关系

线程层级是逻辑抽象，运行时由硬件调度：

  • Block → SM：一个 block 被整体分配到某一个 SM 上执行，且不会跨 SM。这是 block 内线程能共享 shared memory的物理基础——它们都在同一个 SM 上。

  • 一个 SM 可同时驻留多个 block（受资源限制），block 之间无执行顺序保证，也无法直接同步。

  • Grid → 整个 GPU：block 被调度器分发到各个 SM。block 数量通常远大于 SM 数量，SM 执行完一个 block 后再接下一个。

1.3 Warp（线程束）

GPU 实际调度的基本单位，由 32 个线程组成。block 进入 SM 后即被拆成若干 warp。同一 warp 的所有线程以 SIMT 方式执行——同一时刻执行相同指令，各自操作不同数据。

线程数对齐：block 线程数应为 32 的整数倍，否则最后一个 warp 有空闲线程，浪费资源。

线程束分歧（Warp Divergence）

同一 warp 内线程因条件分支走向不同路径时，硬件串行执行各分支，效率降低。

1.4 Block 调度

资源限制

一个 SM 上能同时驻留多少个 block，取决于每个 block 消耗的资源与 SM 的总资源：

  • 寄存器：SM 寄存器总数固定，每线程寄存器数 × block 线程数 越大，能驻留的 block 越少
  • Shared Memory：SM 的 shared memory 总量固定，每个 block 申请越多，能驻留的 block 越少
  • Block / Warp 数量上限：SM 对常驻 block 数和 warp 数有硬件上限

二. GPU 内存层级

GPU 内存按容量、延迟、作用域分层，理解层级是优化的前提：

按从快到慢排列：

内存类型 作用域 速度 说明
寄存器 Register 单线程 最快 线程私有，数量有限，超出则溢出到 local memory
Shared Memory Block 很快 片上，block 内共享，与 L1 cache 共用同一块物理空间
L1 / L2 Cache SM / 全局 快 硬件自动缓存 global memory 访问，程序无法直接控制
Local Memory 单线程 慢 名为 local，实际在 global memory（DRAM）上，寄存器溢出/大数组时使用
Global Memory 全 Grid + Host 最慢 显存（DRAM），容量最大，是主要瓶颈，访问需注意合并

关键点

  • 越靠近计算单元越快、越小：register > shared > L1/L2 > global，延迟可差几十到几百倍。

  • Shared memory 与 L1 共享物理空间，部分架构可配置二者比例（cudaFuncSetAttribute）。

三. CUDA Stream

CUDA Stream 是 GPU 上按顺序执行的操作队列。同一 stream 内操作严格串行，不同 stream 间可并发执行。不指定 stream 时，所有操作进入 default stream。

为什么使用多个 Stream

  • 计算与传输重叠：一个 stream 跑 kernel，另一个同时做 Host-Device 内存拷贝
  • 多 kernel 并发：单个 kernel 占不满 SM 时，多 stream 的 kernel 可同时执行
  • 流水线化：大任务拆分到不同 stream，形成流水线

四. Tensor Core

加速原理

Tensor Core 将矩阵乘加变成一条硬件原子指令，一个周期内执行多次 FMA，吞吐远超标量 CUDA Core。使用方式包括 WMMA API、cuBLAS、混合精度训练等。实际瓶颈通常在 memory bandwidth 而非 compute。

混合精度（Mixed Precision）

混合精度不是指代码中既有 FP16 又有 FP32，而是硬件算子层面：

  • 输入/输出用 FP16，减少显存占用和带宽压力
  • 中间累加用 FP32，保证精度不过度损失

五. GEMM 常见优化

矩阵乘法 C = A × B 是深度学习和 HPC 的核心运算。

5.1 访存合并（Memory Coalescing）

一个 warp 中所有线程同时访存时，若访问连续的全局内存地址，硬件会合并成一次 DRAM 访问，大幅提高带宽利用率。

GEMM 中用 threadIdx 对应矩阵最内层连续维度（列索引），确保同一 warp 读取连续地址。

5.2 Shared Memory

一个 block（32×32 线程）计算 C 的 32×32 子块时，同一行 32 个线程都需要 A 的同一行，同一列 32 个线程都需要 B 的同一列。没有 shared memory 时，数据被重复读 32 次。

Tiling 策略

Shared memory 是 block 私有的片上缓存，延迟极低但容量小，需分块加载：

  1. 将 A、B 的一个小 tile 从 global memory 加载到 shared memory
  2. 线程从 shared memory 读取，完成部分点积
  3. 同步后加载下一个 tile，重复直到完成

每个数据只从 global memory 读一次，block 内所有线程共享。

5.3 Bank Conflict

什么是 Bank

Shared memory 划分为 32 个 bank，每个 bank 宽 4 字节，地址循环分配。一个 warp 的 32 个线程可同时访问 32 个不同 bank，一周期完成。

冲突场景

同一 warp 中多个线程访问同一 bank 的不同地址时产生 bank conflict，硬件串行化，延迟翻倍。

场景 访问模式 结果
无冲突 线程 i 访问 shared[i] 完美并行
2-way conflict 线程 i 访问 shared[2*i] 线程 0 和 16 撞 bank 0，需 2 周期
Broadcast 多线程访问同一 bank 的同一地址 硬件广播，无惩罚

可通过 NVIDIA Nsight Compute (NCU) 检测。

解决方法一：Padding（传统）

声明时多加一列，打破对齐模式：

每行 33 个 float，地址 row * 33 + col，33 mod 32 = 1，每行偏移一个 bank，32 行覆盖 32 个不同 bank。

缺点：浪费空间，tile 尺寸变化时需重新调整。

解决方法二：Swizzle（现代）

CUTLASS、cuBLAS 等主流库采用的方式，对地址做位运算变换，零空间浪费地消除 bank conflict。

核心思想：将列地址的 bank 选择位与行索引做 XOR，使不同行的相同列偏移映射到不同 bank：

第 0 行 bank 位不变，第 1 行 XOR 1，第 2 行 XOR 2……不同行同列被同时读取时，各线程命中不同 bank。

Swizzle 优势：

  • 零空间浪费，shared memory 利用率 100%
  • 数据紧凑，不影响向量化加载（LDS.128）
  • 通用性强，无需针对 tile 大小手动调整
