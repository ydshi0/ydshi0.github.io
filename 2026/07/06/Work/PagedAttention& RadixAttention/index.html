PagedAttention & RadixAttention
eric
  2026-07-06   2026-07-06
  • Work

一. 动机

朴素实现把每条序列的 KV 存成一整块物理连续显存：

  1. 必须按 max_length 预留：不知道要生成多长，只能按最大长度占满 。实际用不到的部分全浪费，内部碎片多。
  2. 无法复用：两条请求前缀即使完全相同，各自是独立连续缓冲区，KV 无法复用。

「之前必须物理连续」—— 早期把 KV 开成 [batch, heads, max_seq_len, head_dim] 的大张量，kernel 用 cache_base + t*stride 这种线性映射直接按位置 t 取地址，所以同序列必须连续。PagedAttention 就是把这个线性映射换成一次查表（block table）的间接映射。

二. PagedAttention

2.1 类比操作系统分页

操作系统 PagedAttention
虚拟地址空间（连续） 一条序列的 KV（逻辑连续）
物理页 page KV block（固定 block_size 个 token，如 16）
页表 page table block table（每条请求一行）
虚拟页号 → 物理页帧 逻辑块号 → 物理块号

2.2 vLLM 里的两个结构：block table 与 slot mapping

源码 vllm/v1/worker/block_table.py

block_table = 页表本体，形状 [max_num_reqs, max_num_blocks_per_req]。第 r 行第 i 列存的是「请求 r 的逻辑块 i 对应哪个物理块」。它是每请求持有、append-only、长期存在的——序列每长一个 block 就往行尾追加一个物理块号

slot_mapping= 本次 batch 里每个新 token 要写入的物理显存槽位，是一个一维数组 [max_num_batched_tokens]。它是每个 batch 现算现用的临时产物。

为什么需要 slot mapping？ 读 KV 和写 KV 都要把「逻辑位置」翻译成「物理地址」（都得查 block_table），差别在于这层翻译是循环里现算，还是预先算成一张表。而这取决于两边访问的规模：

  • 读（= attention 计算）：query 要和全部历史 K/V 做注意力，kernel 本来就得一块块遍历，遍历时现查即可，预计算反而是一张又大又没必要的表。
  • 写（= 存新算出的 K/V）：一步只新增很少的 token（decode 时每条序列就 1 个），还被跨请求打平成一个一维张量。预先算出「第 j 个 token → 物理 slot」只是一张很短的数组，成本极低。

2.3 好处

  • 不需要 max_length：按块（16 token）粒度用多少给多少，浪费 < 一个 block。
  • 可复用：不同请求的 block table 行里可以填同一个物理块号——这是复用的物理基础。
  • 动态无搬移：抢占、swap、beam search 分叉都只改 block table 指针，不搬 KV 数据。

四. RadixAttention / Prefix Caching

RadixAttention 是 SGLang 的叫法，vLLM 对应 Automatic Prefix Caching

radix tree（基数树） = 压缩版的前缀树（trie）：普通 trie 一个节点存一个字符，radix tree 把「只有单一孩子的一串节点」合并成一条边，从根到某节点的路径就代表一段公共前缀。

思想一致：共享前缀的请求共享物理块。vLLM 不建前缀树，而用「带父哈希的块哈希链 」等价实现。

  • 哈希链即树路径：块哈希 = hash(块内容 + 父块哈希)。前缀相同则哈希链前若干项逐个相等——对应radix tree 从根到某节点的一条路径。

  • 匹配最长公共前缀：新请求逐块比对哈希，命中就继续、不匹配即停；命中的块直接被新请求的 block table 引用、ref_cnt++。

  • 缓存与淘汰：块写满时登记进哈希表；ref_cnt>0 不可淘汰，降到 0 回 LRU 队列。当需要新块而无空块时，从队头取最久未用的块淘汰、覆盖其内容。若在用的块也腾不出（全 ref_cnt>0），则转为抢占：丢弃某条请求和它的KV。
