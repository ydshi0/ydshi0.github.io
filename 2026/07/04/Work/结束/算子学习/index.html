〇、概述

大模型推理的两大计算模块是 Attention 和 MoE，对应的高性能算子是 FlashAttention 和 Fused MoE。本文梳理这两个算子的核心思路，以及它们在 serving 场景下的泛化——Flash-Decoding 与 FlashInfer。

一、Online Softmax

1.1 问题

Softmax 是行内全局操作：

softmax(x)i=exi−m∑jexj−m,m=maxjxj

要算任何一个位置的输出，都得先知道整行的 max 和 sum。朴素做法三遍扫描（求 max → 求 sum → 算输出），与分块计算天然冲突。

1.2 解法：max、sum、输出 O 一起在线维护

遇到更大的 max，就用同一个乘子把已算的分母和输出一起"缩到"新尺度。

记已扫描部分的当前状态为 (m, ℓ, O) （分别是当前最大值、当前分母、当前未归一化输出）。新来一个 block，令其注意力分数为 S 、对应 value 为 V ，更新后的状态记为 (m′, ℓ′, O′) 。

维度约定（针对单个 query token、且在单个 head 内；本 block 有 Bkv 个 KV token， d = head_dim，即每个头的维度）：

  • 输入的单个 query 向量 q∈Rd ，输出 o∈Rd （进来是 d 维，出去还是 d 维）
  • S∈RBkv （1D， q 对本 block 各 key 的分数，由 q⋅K⊤ 得到）， m, m′, ℓ, ℓ′ 均为标量
  • V∈RBkv×d （2D 矩阵）， O, O′∈Rd （1D，就是上面 o 的在线累积中间态）
  • eS−m′⋅V 是向量×矩阵（ [Bkv]×[Bkv,d]→[d] ）

（batch_size, num_heads）怎么拆开：每个 (batch, head) 组合当成一个独立任务，分给不同的 thread block / SM 去并行跑，互不干扰。

q 之间也是并行的：由于每个 qi 的输出 oi 只依赖它自己和全部 K/V、和别的 q 无关，所有 query token 可以同时算（FA-1 只并行到 (batch, head) 这一层；FA-2 把 query 维也切出来分给不同 thread block）

thread block ：CUDA 把线程分三层组织：grid → thread block → thread。一个 thread block 是一组线程，会被整体调度到某一个 SM 上执行、共享该 SM 的 shared memory；不同 block 相互独立、不能直接通信。

m′=max(m, rowmax(S))ℓ′=em−m′⋅ℓ+∑eS−m′O′=em−m′⋅O+eS−m′⋅V

同一个因子 em−m′≤1 同时修正 ℓ 和 O 。扫完所有 KV block 后 O←O/ℓ ，一遍扫完。

二、FlashAttention-1（简述）

  • 循环：外循环 KV，内循环 Q。后向用重计算恢复 S/P。

  • 效果：显存 O(N2)→O(N) 。

  • 痛点：外循环 KV → 多个 thread block 要跨 SM 通信更新同一个 Oi ；

三、FlashAttention-2（重点）

三个关键改动，核心目标：喂饱 tensor core + 更好并行。

外循环换成 Q

每个 thread block 认领一段 Qi ，内循环扫所有 KV。每个 block 独立维护 mi,ℓi,Oi ，无需跨 block 通信。HBM 写次数从 O(N/B⋅N/B) 降到 O(N/B) 。

沿 Q 维额外并行

FA-1 只按 (batch, head) 并行。FA-2 额外沿 sequence 维并行（Q 之间独立）。长序列 + 小 batch 时 SM 占用率大幅提升。

减少非 matmul FLOPs

逐步除 ℓ 推迟到最后，中间只维护未归一化的 O。H100 上 tensor core 是非 tensor core 的 ~16×，减少 non-matmul 立竿见影。

Warp 布局

FA-1 split-K（K 维切给不同 warp，需通信合并）→ FA-2 split-Q（每个 warp 拿整段 Q 行，K/V 共享，warp 间零通信）。

四、FA-3 / FA-4（简述）

FA-3（Hopper H100）

WGMMA + TMA + warp specialization（生产者搬数据，消费者算 GEMM，软流水）。GEMM 与 softmax exp 重叠执行。引入 FP8。

FA-4（Blackwell）

跳过已知 m 不变时的 rescale；每线程处理一整行省 shuffle；两个 softmax warpgroup 显式同步。

五、 Fused MoE

一个完整的 MoE FFN 层：router 把每个 token 路由给若干 expert，各 expert 用自己的权重做 FFN，再加权求和。

Fused MoE 把这整条链压进一个（或极少）kernel、中间结果不落 HBM。

5.1 Grouped GEMM（MoE 的核心计算）

router 把 token 分给不同 expert。Grouped GEMM 并行算所有 Ye=Xe⋅We ——就是把一批"形状不同"的 GEMM 打包进一个 kernel 做。

做法：按 expert id 把 token 排序拼接，用 offsets 标边界，kernel 按 offsets 切 tile、tile 不跨 expert。3 个 expert，分到 5/3/7 个 token，block_size=2：

与普通 GEMM 的区别：tile 内部是标准 GEMM，但要按 expert_id 索引不同权重（权重堆叠为 [E, d, d_out]）。

5.2 permute / unpermute

上面"按 expert 排序拼接"这一步没那么理所当然。Router 是一个学习到的网络，每个 token 路由到哪个 expert 是动态、不可预测的，同一 expert 的 token 散落在 batch 各处；而 grouped GEMM 要求同一 expert 的 token 在内存里连续排列才能切 tile。所以要：

  1. Permute：按 expert id 把 token 重排，让同一 expert 的 token 相邻
  2. Grouped GEMM：在重排后的连续数据上按 offsets 切 tile 计算
  3. Unpermute：算完把结果恢复到原始 token 顺序（后续层期望原始顺序）

六、变长处理：一个统一的套路

MoE 的 grouped GEMM 和 FA 处理变长 batch 用的是同一个套路：

核心：不 padding，拼接 + offset 数组 + tile 不跨段。

它和 5.1 的 grouped GEMM 一一对应（cu_seqlens ↔ offsets，请求 ↔ expert）。唯一的不同在 tile 内部：FA tile 做 online softmax + 沿变长 KV 滑动；grouped GEMM tile 做标准 GEMM + 按 expert_id 索引权重。

七、FlashInfer

FA 是"高性能函数"，FlashInfer 是"高性能运行时"：它内含 FA 级别的 attention kernel，但额外解决 serving 场景才有的三个问题。

7.1 paged KV cache

Serving 时每个请求长度不同且动态增长，KV cache 常用 paged 格式（切成固定页、page table 索引）。代价是 KV 物理上不连续，普通 FA kernel 要先拷成连续 tensor 才能算。FlashInfer 的 kernel 直接读 paged 格式，省掉拷贝。

怎么做的：加一层间址

load 每个 KV tile 前先查一次 page table 拿到物理页号再算地址，tile 内部计算完全不变。这套间址统一成 block-sparse 抽象（paged、变长、causal mask 都只是不同索引）。代价是多一次查表，但省掉 decode 每步的整块拷贝。

7.2 变长负载均衡

同一 batch 里序列长度差距很大（100 vs 100k）。decode 阶段每个请求只有 1 个 query token，query-tile 维度塌了，只剩 (batch, head) 可并行——每个 (请求, head) 一个 thread block、串行扫完自己全部 KV。于是 100 的 block 秒完空闲、100k 的 block 独占一个 SM 扫上千步，成为瓶颈。

集成 Flash-Decoding

既然 query 维切不动，就沿 KV 维再切一刀——把长序列的 KV 切成多个 chunk 丢给不同 SM 并行扫，各产出局部 (m, ℓ, O)，最后用 online softmax 合并（和 block 间在线累积同一套数学）。长序列的活儿由此摊到多个 SM，SM 不再干等。

7.3 attention 变体统一

causal、sliding-window、GQA、MQA 等变体太多。FlashInfer 用参数化 CUDA 模板 + JIT：首次遇到新配置即时编译并缓存复用，不用为每种组合手写 kernel。
