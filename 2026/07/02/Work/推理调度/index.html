一、推理的两个阶段

Prefill

  • 并行处理整个 Prompt，计算所有输入 Token 的 KV Cache，并生成第 1个输出 Token。
  • Compute-bound，计算量大，能够充分利用 GPU 算力。

Decode

  • 基于 KV Cache 自回归生成 Token，每个 iteration只能生成一个 Token。
  • Memory-bound，大量时间花在读取 KV Cache，GPU算力难以打满，通常依靠 Batch 提高利用率。

二、调度的目标

推理调度（Scheduling）的目标，是决定每个 iteration GPU应该执行哪些请求，在 TTFT（Time To First
Token）、**ITL（Inter-Token Latency）**和系统吞吐量（Throughput）之间取得平衡。

  • TTFT：从请求到达，到生成第一个 Token 的时间

  • ITL：相邻两个输出 Token 的时间间隔

  • Throughput：单位时间内系统生成的 Token 数

三、调度策略

3.1 Static Batch

早期推理引擎采用静态 Batch。

先收集一批请求，再统一执行；只有当这一批全部完成后，GPU 才能处理下一批。

因此，短请求必须等待长请求结束，GPU 经常因为等待而闲置，吞吐较低。

3.2 Continuous Batching

Orca 提出了 Continuous Batching。它将调度粒度从整个请求降低到每个 iteration。

Scheduler可以重新决定下一轮 Batch的组成。完成的请求立即退出，等待队列中的新请求也能立刻补入，无需等待整个Batch 完成。

3.3 新问题：Prefill 阻塞 Decode

为了降低新请求的 TTFT，调度器通常会优先执行 Prefill。

例如，A、B 正在稳定 Decode（每步约 10 ms），此时来了两个 32K Prompt的请求 C、D。如果立即执行它们的 Prefill，GPU 可能连续占用数百毫秒，导致A、B 无法继续 Decode，ITL 明显升高。

3.4 Chunked Prefill

将一个长 Prompt 划分为多个固定大小的 Chunk（例如 2048Token），每个 iteration 只处理一个 Chunk。

两个 Chunk 之间，调度器可以插入其他请求的 Decode，使Prefill 与 Decode 在同一个 Batch 中执行。

收益：

  • Decode 不再被长时间阻塞，ITL 更平滑。
  • Compute-bound 与 Memory-bound 混合执行，提高 GPU 利用率。

代价：

  • 长 Prompt 的 TTFT 会略有增加，因为需要分多次完成 Prefill。

本质上，是牺牲少量单请求 TTFT，换取整体吞吐和在线请求更低的延迟。

四、Chunked Attention 如何计算

每一步只做两件事：

  1. 对当前 chunk 的新 token 算 Q、K、V（Q 只用一次，K/V 追加进 cache）
  2. Attention 的 K/V 范围 = cache 里的历史 + 当前 chunk 新算的；chunk 内部仍是 causal mask，对历史 KV 全部可见
