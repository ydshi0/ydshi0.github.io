EPLB：专家并行负载均衡

Expert Parallelism Load Balancer
仓库：https://github.com/deepseek-ai/EPLB（核心仅 eplb.py 一个文件）

一、背景

1.1 专家并行（Expert Parallelism, EP）

MoE（Mixture of Experts）把 FFN 层换成 N 个并列的「专家」（每个专家就是一个 FFN）。专家并行把不同专家分散到不同 GPU 上。

1.2 MoE 的负载不均衡现象

理想情况下每个专家被选中的概率应该差不多，但实际推理中专家负载是高度倾斜的：

  • Router 的选择由数据分布决定，某些专家是「热点专家（hot expert）」，被大量 token 选中；
  • 另一些专家几乎没人用（冷专家）；
  • 倾斜随输入内容、任务类型动态变化，并不固定。

后果：整个 EP 组的速度由最忙的那张 GPU决定。一张卡上的专家被挤爆，其它卡都在空等它的 All-to-All 结果，GPU 利用率严重下降。

二、EPLB 的核心思想

2.1 冗余专家（Redundant Experts）

对热点专家做复制：一个逻辑专家可以在多张 GPU 上有多个副本，token 被分摊到这些副本上，从而把热点的负载切开。

概念 含义
逻辑专家 (logical expert) 模型里真实存在的专家，共 num_logical_experts 个
物理专家 / 副本 (physical expert / replica) 实际部署到 GPU 上的实例，共 num_replicas 个
冗余专家数 num_replicas - num_logical_experts

2.2 启发式打包（Heuristic Packing）

把这些物理专家（含副本）尽量均匀地塞进各张 GPU，使每张 GPU 的预估负载尽量接近。

负载怎么估？

常用历史统计的滑动平均（moving average）作为 weight。

2.3 group-limited routing 降低跨机通信

DeepSeek-V3 用了 group-limited expert routing：专家分成若干 group，一个 token 的路由被限制在少数几个 group 内。EPLB 因此尽量把同一个 group 的专家放在同一台 node 上，让 All-to-All 尽量走机内高速网络（NVLink），减少跨机流量。

路由工作流程（这套路由必须在训练时就生效）：

  1. 每个 token 对 256 个专家算亲和度分数；
  2. 给每组算「组分数」（V3 用组内 top-2 分数之和）；
  3. 选分数最高的 topk_group=4 个组；
  4. 只在这 4 组（4×32=128 个专家）里再选 top-8。

三、算法逻辑：复制 + 装箱（核心两步）

EPLB 主要就是两步，复制 + 装箱。这两步所有模型、所有阶段都适用（即「全局策略」）

摊薄负载 = 专家负载 / 副本数（weight / logcnt）。 复制后 token 分摊到各副本，真正决定单卡压力的是摊薄负载——复制时挑「摊薄后最重的」加副本，装箱时也按摊薄负载分。

复制 replicate_experts

把热点专家复制成多个副本（min-max，让最大副本负载最小）

装箱 balanced_packing

把专家均匀分到各 GPU（贪心，类似 LPT 最长作业优先；每个 GPU 专家数相等、总负载尽量接近）

层次化（仅 prefill）

如果模型有 group-limited routing（DeepSeek ）且处于 prefill（EP 小），就在两步前面加一步：

唯一目的：让同 group 专家落在同一台机器，token 的 All-to-All 尽量走机内 NVLink，省跨机通信。

为什么只有 prefill 加？

prefill 大 batch、通信量大 → 省跨机收益最大，值得加；

对外入口 rebalance_experts

输出三件套：phy2log（副本→专家）/ log2phy（专家→副本）/ logcnt（各专家副本数）。
