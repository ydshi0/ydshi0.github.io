一、从 EAGLE-3 到并行 Draft

投机解码一轮的平均单 token 延迟可以写成：

L=Tdraft+Tverifyτ
  • Tdraft ：生成 draft token 的时间；
  • Tverify ：目标模型验证 draft 的时间；
  • τ ：一轮实际产出的 token 数，即平均接受长度（包含目标模型额外生成的 bonus token）。

因此，加速投机解码有三个方向：draft 更便宜、draft 更准、verify 更高效。

EAGLE-3 依然是自回归 drafter，需要逐个生成 draft token。它能利用已经采样的前缀，后续 token 之间的依赖建模较强，但 draft 延迟会随长度近似线性增加。为了控制成本，EAGLE-3 通常只能采用较浅的 draft 网络。

DFlash 和 DSpark 走的是另一条路线：先把主要计算改成整块并行生成，再解决并行生成带来的质量和验证浪费问题。

整体演进关系可以概括为：

  1. EAGLE-3 用完整的自回归依赖换取稳定后缀，但多次串行 draft forward 限制了网络深度和 block 长度。
  2. DFlash 用 target feature KV injection 和一次并行 block forward 提升 draft 速度与容量，代价是后续位置看不到前面实际采样的 token，存在 suffix decay。
  3. DSpark 在 DFlash backbone 后增加轻量 Markov Head，用很少的串行计算补回块内依赖。
  4. Confidence Scheduler 再根据 token 存活概率和硬件负载动态裁剪验证长度，减少高并发下的无效 target compute。

因此，DSpark 的重点不只是 drafter 更准，而是把 draft 生成、接受概率估计和 serving 调度放进同一个优化目标中。

二、DFlash：用 Block Diffusion 并行起草

核心洞察

目标模型在生成当前 token 时，其中间 hidden states 已经包含了对多个未来 token 的信息。与其让一个小模型从 token 上下文重新推理，不如把目标模型的内部特征直接注入 drafter，再让 drafter 一次预测整个 token block。

目标模型特征注入

DFlash 从目标模型由浅到深选取若干层 hidden states。对于每个已经确认的 token，将这些层的特征拼接并投影到 draft hidden size，得到该 token 的 target context feature。

推理时只传入本轮新增的 target features，记为 Cnew ；此前已经确认的前缀特征已投影并保存在 drafter 的 Kcache,Vcache 中。当前 draft block 的 hidden states 记为 Z ，则每层 attention 实际计算为：

Q=ZWQ,K=[Kcache;CnewWK;ZWK],V=[Vcache;CnewWV;ZWV]

所以 context 长度并不固定为 1：

  • 第一轮 speculative decoding 会把整个 prompt 的 target features 写入 draft KV cache；
  • 后续一轮只追加上一轮最终提交的 accepted_draft_tokens + 1 个 target features；只有一个 draft token 都没接受时，新增长度才是 1；

每一轮中，当前 block 的每个 draft query 都会 attend 到前面所有已确认 token 的 K/V，再加上当前 block 的 K/V。

一次前向生成整个 Block

每轮验证结束后，目标模型会额外生成一个 bonus token。DFlash 把它作为 anchor token，在其后放置一组 mask token：

mask 位置之间使用双向注意力，整块 token 在一次 draft 前向中完成预测。这里称为 block diffusion，是因为 drafter 学习从 masked block 恢复干净 token；但它不是在推理时执行很多轮去噪，而是只做一次并行去噪。Draft 成本因此不再随 block 长度线性增长

局限

虽然 mask positions 在网络内部可以双向交互，但它们看到的仍然是 mask 表示，后一个位置无法条件化于前一个位置最终采样出的具体 token。

核心思路

DSpark 同时解决两个问题：

  1. Draft 质量：在 DFlash 式并行 backbone 后增加极轻量的 sequential head，补回块内 token 依赖。
  2. Verify 效率：预测每个 draft token 的接受概率，根据请求难度和系统负载动态决定验证多长。

四、Semi-Autoregressive Generation

并行阶段

DSpark 的 parallel backbone 基于 DFlash：从目标模型抽取多层 hidden states，通过 KV injection 注入每一层 drafter，再对整个 block 做一次双向注意力前向。

论文实现对 DFlash 做了一个小改动：把所有预测位置向左移动一格，让 anchor 位置的输出直接预测第一个 draft token。

这个改动不会引入新的串行计算： γ 个 base logits 仍在一次 backbone forward 中并行产生，只是少计算了一个 mask position。随后 Sequential Block 才使用 anchor x0 修正 U1 、采样 x1 ，再依次修正后续位置。

这一阶段负责主要的语义建模，计算量较大，但完全并行。

串行阶段：Sequential Block（默认是 Markov Head）

论文中的 Sequential Block 是一个统称，不是额外的 Transformer block。它有两种实现：默认的 Markov Head，以及可选的 RNN Head。

并行 backbone 先一次性输出整块 hidden states h1,…,hγ ，再通过共享 LM head 得到 base logits U1,…,Uγ 。Sequential Block 随后从左到右运行；在第 k 个位置，它的输入和输出为：

内容
输入 1 当前位置的 base logits Uk∈RV
输入 2 前一个实际采样的 token xk−1 ；当 k=1 时，它就是上一轮目标模型生成的 anchor token x0
输出 1 词表维度的转移偏置 B(xk−1,⋅)∈RV
输出 2 修正后的 logits U~k=Uk+B(xk−1,⋅) ，以及从中采样出的 token xk

新采样出的 xk 会作为第 k+1 步的输入，因此 block 内形成一条很轻的串行依赖：

所以它不是 MLP：中间没有激活函数、归一化或多层非线性变换，只有一次 embedding lookup 和一次线性投影。它等价于用两个低秩参数矩阵近似一个完整的 V × V token 转移表。

严格来说，Markov Head 只负责产生 [B, V] 的 transition bias；将它加到当前位置的 base logits 并采样 token，是外层 sequential sampling loop 完成的。由于 bias 只取决于前一个 token，这仍然是一阶 Markov 依赖。

采样仍然要从左到右进行，但每一步只包含一次 embedding lookup、低秩投影和 logits 相加，不需要重新运行 Transformer backbone。论文中把 block 长度从 4 扩展到 16，相比 DFlash 增加的整轮延迟只有约 0.2%～1.3%。

论文也实现了 RNN Head：除了 Uk 和 xk−1 ，它还显式接收当前位置的 hk 与上一步递归状态 sk−1 ，因此可以累积完整块内历史。RNN 在长 block 上略好，但收益有限、部署更复杂，所以默认采用 Markov Head。

五、Confidence-Scheduled Verification

更长的 draft block 不等于更快。如果每个请求都固定验证完整 block，高并发下目标模型需要处理大量最终会被拒绝的尾部 token，反而占用 batch capacity。

Confidence Head

Confidence Head 为 block 中的每个 draft token 预测一个接受置信度。越靠后的 token 只有在前面全部通过时才可能被接受，因此调度器关注的是整个前缀能存活到该位置的概率，而不是孤立地看单个 token。

原始 confidence 可能过度自信，DSpark 使用 Sequential Temperature Scaling（STS）做离线校准，使置信度更接近真实接受率。STS 只校准分数，不改变 draft token。

动态调整验证长度

DSpark 每轮仍然生成完整 draft block，但不一定把整块都送给 target。调度器结合两类信息，为每个请求选择不同的验证前缀长度：

  • 请求本身的置信度：高置信度前缀更长，值得继续验证；低置信度后缀大概率会被拒绝，提前裁掉。
  • 当前系统负载：低负载时多验证几个 token 的代价较小；高负载时 target batch capacity 更宝贵，需要缩短验证窗口。

引擎会提前测量不同 batch size 下的实际速度，调度器据此判断“多验证一个 token 带来的预期收益，是否值得增加 target 计算”。因此它不是使用固定 confidence 阈值，而是随硬件和实时负载变化。

调度器只决定验证多长，最终接受哪些 token 仍由 target model 的 speculative sampling 规则决定，所以不会改变目标模型的输出分布。

生产实现还需要把不同请求的变长前缀紧凑打包，通过 ragged verify 和 CUDA Graph 执行，确保裁掉 token 后确实减少计算，而不是又被 padding 补回固定长度。

六、实验结果

在关闭 confidence scheduler、固定验证长度的离线实验中，DSpark 在数学、代码和对话任务上都优于 EAGLE-3 与 DFlash，并且这一趋势在不同规模的 Qwen3 和 Gemma 模型上保持一致。这说明收益确实来自半自回归 drafter 本身，而不是调度策略。

逐位置结果进一步印证了前面的分析：EAGLE-3 的首 token 受浅层模型容量限制，但后缀比较稳定；DFlash 的首 token 更强，却会随位置出现明显的 suffix decay；DSpark 同时保留了较高的首 token 接受率和更稳定的后缀接受率。

在线上高并发场景中，confidence scheduler 会在负载较低时分配更长的验证窗口，在负载升高时主动裁剪低置信度后缀。相比固定长度验证，它减少了无效 target compute，并改善了吞吐与单请求生成速度之间的 Pareto frontier。
