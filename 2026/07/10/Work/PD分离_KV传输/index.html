PD 分离的 KV 传输
eric
  2026-07-10   2026-07-10
  • Work
  • KV 传输

本文重点：不同并行配置下，KV 如何拆成 (P rank, D rank) 点对点传输。

一、核心模型：rank 传输边

本文讨论的主流路径中，KV cache 在 (P rank, D rank)之间点对点传输，不经过中心节点

除点对点外，也存在中心化 / 池化的 KV 传输方案：P 把 KV 写入一个共享 KV store，D 再从中读取，两侧不直接连接。典型如 Mooncake Store、LMCache 的远端后端（Redis / 对象存储 / 分布式 KV 池）。它的优点是天然支持跨请求前缀复用、P/D 解耦和弹性扩缩，代价是多一跳存储、时延更高。

一次传输分为两步：

  1. 建边：选择 P/D 实例，根据 DP/TP/CP/PP 布局计算 rank 映射和 KV 切片。
  2. 执行：每对 (P rank, D rank) 直接搬运自己的 KV，并汇总完成状态。

传输粒度：不是「一个 block 发一次」，而是一条请求的整份 KV 在一次调用里批量发出。注意「批量」指一次 API 调用提交一串 descriptor，不是把数据先拼成连续内存——KV 仍留在原来分散的 paged 地址上，靠 RDMA 的 scatter-gather 逐段搬。

RDMA scatter-gather 是什么？ RDMA scatter-gather 指网卡能接收一份内存段清单（SGL），在一次操作里搬运多个互不相邻的段：发送端从多个分散源地址「gather」，接收端写入多个分散目的地址「scatter」。分散的 block 地址不必先拷成连续内存就能一次发。

常见传输后端：

  • NIXL：NVIDIA 的传输抽象层，通过统一 API 屏蔽多种底层插件，按运行时配置选择实际传输路径。
  • Mooncake Transfer Engine：面向 PD 分离的高性能传输引擎，以 RDMA 为主在多机多卡间点对点搬运 KV，并可对接分级存储做 KV 缓存池。
  • LMCache：KV 缓存复用层，负责 KV 的存储、卸载（CPU/磁盘）与跨请求前缀复用，通常搭配上面的传输后端而非自己承担点对点搬运。

二、KV 在显存里怎么存：哪些维度连续

以 SGLang MHA 为例，每层的 K、V 各是一块独立张量，共 2 × layer_num块：

单块张量内是行主序，一个 token slot 由 head_num × head_dim × dtype_bytes 三维决定；一个 block（vLLM 叫 block、SGLang 叫 page）= 连续 page_size 个 token slot——「block size = 16」指的就是一块放 16 个 token：

关键：layer 和 K/V 都是各自独立复制的——第 L 层和 L+1 层、K 和 V，地址互不相邻。所以一个模型存 16 个 token 的完整 KV，需要 layer_num × 2（K、V）个 block。

于是单条边的 descriptor 数 = 连续段数 。而取 head 子区间（异构 TP）还会打破页内连续性：head 不是最内维，只要某几个 head，每个 token 的 head slice 之间就隔着完整的 head stride，只能逐 token 下 descriptor (详见第六节）。

三、地址和 block 映射如何建立

3.1 SGLang：P WRITE

SGLang Mooncake/NIXL 的常见流程是：

  1. P ranks 通过 register_to_bootstrap() 注册各自的 DP/TP/CP/PP rank 和通信地址。
  2. D 查询 P 的并行布局，在本地通过 _resolve_rank_mapping() 算出目标 P ranks。
  3. D 用 _register_kv_args() 向相关 P ranks 注册 KV pool 基址等连接级信息。
  4. D 用 send_metadata() 发送本请求的 kv_indices 和完成计数。
  5. P 完成 prefill 后，根据两侧 block IDs 和内存布局向 D 发起 WRITE。

四、并行配置下如何选 (P rank, D rank)

一个 rank 可以由一组坐标唯一确定：

选边就是逐维度确定这组坐标，并算出每条边搬运哪一段 KV。 各维度的作用可以分工来看：

维度 决定什么 对边的影响
服务副本 + attention DP 请求落到哪个 engine、哪个 DP worker 先把范围缩小到一个具体的 rank 网格
attention TP KV head 怎么切分 决定边的条数和每条边的 head 切片
CP 谁来发送完整 KV 默认只有 CP rank 0 参与；可选让各 CP rank 分摊
PP KV 按 layer 切分 决定每条边搬运哪些 layer

下面按这个顺序展开。约定：

异构 attention TP 映射要求 P、D 之间存在实现支持的整除关系（P | D 或 D | P）。

两个贯穿全篇的计数器：

  • required_dst_info_num：一个 P rank 要等多少个 D rank 注册后才能开始发送。
  • required_prefill_response_num：一个 D rank 要等多少个 P rank 完成后，本地 KV 才算齐。

4.1 服务副本与 attention DP

选边前要先经过两层定位：

  • 服务副本（部署层）：Router 在多个副本间做负载均衡，选副本确定目标 engine。
  • attention DP（engine 内）：多个 DP worker 处理不同的请求，选完副本后还要确定请求落到哪个 DP rank。

关键点：attention DP 分的是请求，不是单条请求的 KV。 每条请求在 P、D 两侧各自被路由到一个 DP worker，之后 TP/CP/PP 的建边都发生在这一对选定的 worker 内部。所以 DP 影响的是并发请求数（吞吐），而不是一条边怎么切——即便两侧 attn_dp_size 不同也没关系，各自独立选一个 worker 即可，不会把同一条请求的 KV 拆到多个 DP worker 上传输。

4.2 attention TP

SGLang 的 rank 映射只比较两侧的 attn_tp_size，即一个 attention DP/CP group 内实际参与 attention head 和 KV head 切分的 rank 数。本文提到的 TP 均指 attention TP。按 P、D 的整除关系分三种情况：

① P = D：一对一

两侧 head 切分方式相同，每个 D rank 直接接收对应 P rank 的整份本地 KV shard，无需再切 head。

② D > P：一个 P 对多个 D（例如 P=1, D=2）

P0 持有全部 heads，要拆给两个 D。SGLang push 时 P0 向两个 D 写不同切片；vLLM pull 时两个 D 各自从 P0 读不同切片。P0 需先收齐两个 D 的目标 block 信息（required_dst_info_num = 2）。

③ P > D：多个 P 对一个 D（例如 P=2, D=1）

D0 需要完整 KV，而每个 P 只持有一部分 heads，所以要等两条边都完成（required_prefill_response_num = 2）。

head 偏移：P > D 时，每个 P rank 把自己的本地 heads 写入 D rank 中不同的连续位置；D > P 时，多个 D ranks 分别从同一个 P rank 中取不同的 head 子区间。这个偏移就是第六节地址公式里的 head offset。

4.3 CP：默认只由一个 rank 发送

CP 切分上下文计算，但不代表每个 CP rank 的 KV pool 只保存一段序列。当前 Dense MHA 路径会 all-gather K/V，把完整 KV 写入每个 CP rank，因此默认由 CP rank 0 发送就能拿到完整数据——选边时其余 CP rank 可以忽略。

打开 SGLANG_DISAGGREGATION_ALL_CP_RANKS_TRANSFER 后，各 CP ranks 通过 filter_kv_indices_for_cp_rank() 分担不同 pages，D 需等所有参与者完成。它主要用于把传输带宽摊到多个 rank 上。

4.4 PP：按 layer 切分

PP 按 layer 切分 KV，因此它决定每条边搬运哪些 layer：

  • P、D 的 PP size 相等时，对应 stages 一对一传输，各自搬自己那段 layers。
  • D PP=1, P PP>1 时，单个 D 要从所有 P stages 接收不同 layers，required_prefill_response_num 相应乘上 P 的 PP size。

注意 PP 与 attention TP 是正交的：先按 PP 定「哪些 layer」，再按 TP 定「哪些 head」，两者叠加后才是一条边真正搬运的 KV 切片。

六、传输前的 rank 内合并：staging

正常情况下，每条都是整页大块，RDMA scatter-gather 应付得很轻松。只有当异构 attention TP 需要按 head 切片、descriptor 碎成小块时，清单才会长到拖垮传输。

6.1 零散的来源

异构 attention TP（P≠D）下，一条边只需要页内的一个 head 子区间：取某几个 head 时，每个 token 的 head slice 之间都隔着完整的 head stride，无法整页连续拷贝。于是只能对页内每个 token 各下一条 descriptor，descriptor 数增加。descriptor 越多，传输前的地址构造、注册和下发开销越大；prompt 越长、token 越多，这部分开销越大，直接抬高 TTFT。

6.2 staging

staging 的思路是先在本地把零散数据攒成一大块连续内存，再走少量大块传输，用一次显存内的重排换掉成千上万次网络小传输。

三步：

  1. gather（P 侧）：P 用一个 GPU kernel 把散落在各 page 里的 head slices 拷进一块连续的 staging buffer，让它们在显存里首尾相接。
  2. bulk 传输：P/D 之间只对这块连续 buffer 做少量大块 READ/WRITE，带宽打满、descriptor 极少。
  3. scatter（D 侧）：D 收到连续 buffer 后，再用 kernel 把数据散回自己 paged KV 的各个 block 里。

这里的 gather/scatter 是单个 rank 在自己显存内部的数据重排（一次本地 kernel 拷贝），不是 all-gather 那种跨 rank 的 collective 通信。

对应 SGLang 源码 common/staging_buffer.py：staging buffer 与重排 kernel。
