异步调度让「调度 step N+1」与「GPU 执行 step N」重叠以消除 GPU 气泡

一. 调度器

FCFS 原则

每个 step 有一个 token_budget（= max_num_batched_tokens），调度器按 FCFS 顺序先扫 running 队列，再扫 waiting 队列，输出 SchedulerOutput，核心字段是 num_scheduled_tokens: dict[req_id, int]——每个请求本轮处理多少 token。

调度器决定"谁算、算多少"

SchedulerOutput 作用：调度器把它传给 GPU worker，worker 根据 num_scheduled_tokens 为每个请求拼接对应长度的 input_ids，构建 attention mask 和 position ids，然后送入模型做 forward。

Worker 维护一个 token_ids_cpu[max_num_reqs, max_model_len] 数组，每行存一个请求的完整序列 [prompt | output | spec_tokens]。每步从中最后取 num_new_tokens 切片作为 input_ids（之前的 token 已有 KV cache）。

  • prefill：一个 prompt chunk。
  • decode：上一步采样 token + N 个 spec tokens

所有请求的 input_ids 拼成扁平张量送入模型。

    running 中最多只有最后一个请求处于 partial prefill 状态，排在它前面的 decode 请求天然先被调度，等价于 decode 优先。

  • long_prefill_token_threshold > 0，prefill 请求每步最多处理 threshold 个 token

    此时 running 队列中 prefill 可能排在 decode 前面，decode 优先不再严格成立。

二. 异步调度

为什么需要异步调度

GPU forward 一次通常耗时几十 ms，调度本身也需要 ms 级时间。同步模式下二者串行，GPU 每步都有空闲气泡。异步调度让"调度 step N+1"与"GPU 执行 step N"重叠，消除气泡。vLLM（AsyncScheduler）和 SGLang（enable_overlap_schedule）都实现了这一优化，是高吞吐推理的标配。

核心矛盾：调度时不知道执行结果

异步调度下调度器面临根本问题：准备 step N+1 时，step N 的输出尚未返回。

重点：序列实际增长了多少 token → KV Cache block 可能不够

三. Placeholder 机制

vLLM 的解法：用 num_output_placeholders 记录「已调度出去、但结果尚未返回的 token 数」

3.1 核心概念

Placeholder 是一个计数器，定义上表示"已调度但未拿到结果的 token 数"；作用上让 block allocator 据此多占物理 block，等结果回来后释放或确认。

统一调度公式：num_new_tokens = num_tokens_with_spec + placeholder - num_computed_tokens

placeholder 膨胀 num_new_tokens → allocate_slots() 看到更大序列长度 → 多分配 block。

3.2 四个关键字段

字段 含义 更新时机
num_tokens len(prompt + output_token_ids)，已确认序列长度 GPU 结果返回后追加 accepted tokens
num_tokens_with_spec num_tokens + len(spec_token_ids) draft model 写入 spec → 调度消费后清空
num_computed_tokens 调度器认为"算到哪了"的乐观估计 调度后立刻 +=；结果返回时按被拒 token 数 -= 回退
num_output_placeholders 已调度但结果未返回的 in-flight token 数 调度后 +=；结果返回时按本轮消耗量 -=

num_tokens_with_spec vs placeholder 的分工

num_tokens_with_spec 在当前 step 把 spec tokens 调度出去（消费后清空）；placeholder 在下次调度时记住上轮调度了多少还没回来。

假设：prompt 10 tokens 已 prefill 完毕，上一步产出了 t1，进入投机 decode 阶段（异步调度，投机 5 tokens）。

为什么最终 num_computed_tokens(20) < num_tokens(21)？

最后一个 token（bonus token）是 target model 验证时顺带采样的——有 token id 但还没算 KV。下次调度 num_new_tokens = 21 + 0 - 20 = 1，正好为它补上。

3.4 为什么不直接置 0

Placeholder 用 += 累加。异步调度下多个 step 可同时 in-flight，数量由 batch_queue_size（= max_concurrent_batches）决定：无 PP 且开启异步调度时为 2，PP 时为 pp_size。每次结果返回只消耗属于该轮的量，不能置 0，否则会释放其他 in-flight step 的预留空间。
