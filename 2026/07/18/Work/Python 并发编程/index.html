核心组件

Python 主要通过协程、线程和进程实现并发。选择方案时，先判断任务主要在等待 I/O，还是消耗 CPU。

场景 常用方案
网络、RPC、文件等 I/O asyncio、线程池
会释放 GIL 的 C/C++ 扩展 线程池
纯 Python CPU 密集计算 多进程
线程间传递任务 queue.Queue

下文默认讨论启用 GIL 的常规 CPython。

1. 并发基础与 GIL

  • 并发：多个任务在同一时间段内交替推进。
  • 并行：多个任务在同一时刻真正同时执行。
  • 同步：调用方等待操作完成后继续。
  • 异步：操作发起后，调用方可以先处理其他任务。
  • 阻塞：执行线程因为等待而无法处理其他任务。

进程拥有独立地址空间；同一进程内的线程共享内存；协程通常由单个线程中的事件循环调度。

常规 CPython 使用全局解释器锁 GIL，同一时刻通常只有一个线程执行 Python 字节码，因此多线程难以加速纯 Python CPU 密集计算。阻塞 I/O 和部分 C/C++ 扩展会在执行期间释放 GIL，所以多线程仍适合 I/O 或底层计算任务。无论是否启用 GIL，共享可变状态都需要明确的同步协议。

2. 竞态与线程同步

同一进程内的线程共享 Python 对象、模块状态和文件描述符，但各自拥有独立的调用栈和执行位置。

start() 启动线程，join() 等待线程结束。

什么是竞态条件

多个线程并发访问共享可变状态时，如果程序结果取决于不可预测的执行顺序，就出现了竞态条件（race condition）：

两个线程可能同时通过检查，随后都扣减库存。“检查 + 扣减”必须作为一个整体执行，这段不能被其他线程穿插的代码称为临界区。GIL 只约束 Python 字节码的执行，不能保证多步业务操作的原子性。

Lock：保证临界区不被穿插

解决竞态的关键是找出必须整体执行的操作，并让所有相关线程使用同一把 Lock。例如，转账的扣款和入账必须位于同一临界区：

这样，其他遵循相同加锁协议的线程只能看到转账前或转账后的状态。临界区应尽量短，不要在锁内执行 RPC、磁盘 I/O 或 Future.result()；使用多把锁时应保持一致的获取顺序，避免死锁。

6. Queue：在线程间交接任务

queue.Queue 是线程安全的任务队列。生产者通过 put() 放入任务，消费者通过 get() 取走任务；队列为空时，get() 会让消费者休眠，直到有新任务到来。

一个消费者如何工作

get() 表示任务被取走，不表示任务已经处理完成；因此无论成功、失败还是收到 STOP，都要在 finally 中调用 task_done()。调用方可以使用 tasks.join() 等待所有已放入任务都调用完 task_done()。

STOP 是一条控制消息。消费者收到它后退出自己的消费循环和线程，不是退出整个 Python 进程。有多少个消费者线程，通常就放入多少个 STOP，让每个线程都有机会收到停止消息。

限制排队数量

无界队列在生产速度长期高于消费速度时，会不断积压任务，增加内存占用和排队延迟。Queue(maxsize=100) 将等待处理的任务限制在 100 个：队列满时，默认的 put() 会阻塞生产者；使用 put(..., block=False) 则会抛出 queue.Full。

7. multiprocessing 与进程间通信

多进程适合纯 Python CPU 密集任务。每个进程拥有独立的虚拟地址空间和解释器，因此可以利用多个 CPU 核；代价是父子进程中的普通 Python 对象彼此独立，必须显式通信。
