CUDA Graph

  1. CPU 每次向 GPU 提交 kernel 都要走一次 CUDA API 调用，有固定的调度延迟。
  2. CUDA Graph 在预热阶段把所有 kernel 启动、内存操作录制成一张静态图，之后每次 replay 只需一次 API 调用把整张图提交给 GPU，CPU 侧的调度开销从 N 次压缩到 1 次。

torch.compile

  1. 分析算子图，把多个相邻的显存带宽瓶颈算子（elementwise、norm、激活函数）合并成一个 Triton kernel。中间结果不写回 HBM，直接在寄存器里流转，减少 HBM 读写次数。
  2. Triton 编译器会对 tile size 等参数做 auto-tuning，找到当前硬件下的最优配置。

DeepGEMM warmup

DeepGEMM 使用 JIT 编译 CUDA kernel，首次推理时会触发 nvcc 编译导致延迟，需要warmup预编译kernel。

理论

GEMM 问题：C[M, N] = A[M, K] × B[N, K]^T，其中 M 是 token 数（动态），N、K 是模型权重维度（固定）。
切分：M 按 BLOCK_M 切分（候选值 64, 128, 256），N 按 BLOCK_N 切分（候选值 16 到 min(256, N)，步长 16）。
SM 利用率模型：
total_blocks = ceil(M / BLOCK_M) × ceil(N / BLOCK_N)
waves = ceil(total_blocks / NUM_SMs)
utilization = total_blocks / (waves × NUM_SMs)

最后一轮 wave 可能"装不满"所有 SM，造成尾部浪费。不同 M 值改变 total_blocks，从而改变利用率。
Kernel 选择：对所有 (BLOCK_M, BLOCK_N) 组合二重循环，选 waves 最少且最后一轮最满的组合。相同 N、K 下，不同 M 值可能选出不同的最优组合。

VLLM参数配置

relax：

  • 小值覆盖：[1, 2, 4, 8, 16, …, 64]，共 11 个值，覆盖 batch_size 极小时的场景。
  • block_m 倍数：覆盖所有 block_m（64/128/256）的整数倍。因为 num_m_blocks = ceil(M / block_m)，M 每跨过一个 block_m 倍数边界，M 方向的 block 数就多一个，改变total_blocks，从而可能改变最优 (block_m, block_n) 选择，触发新 kernel 编译。
  • wave 转换点：对每组 (block_m, block_n) 候选，反推使 total_blocks = wave × NUM_SMs 的临界 M 值（M = wave × NUM_SMs × block_m / ceil(N / block_n)）。M 跨过这些值时，waves 数从wave 变为 wave+1，DeepGEMM 的第一优先级（waves 最少）可能导致最优 tile 组合切换，触发新 kernel 编译。

full：从 1 到 max_tokens

skip：不预热

VLLM dummy run

用于显存峰值 Profiling

  1. 启动后用最大 batch size（max_num_tokens）跑一次完整的假 forward pass，触发和真实推理完全相同的显存分配路径
  2. 用 memory_profiling 上下文管理器在 forward 前后各拍一次快照，测出三项：模型权重、forward 过程中的激活峰值（torch peak）、非 Torch 内存（NCCL buffer、attention backend buffer 等）

–gpu-memory-utilization

  1. gpu_memory_utilization 在加载权重之前就算定：requested_memory = 总显存 × utilization，这是 vLLM 允许自己使用的总预算上限，与实际用了多少无关
  2. profile run 在预算内量出"已知消耗项"，最终：available_kv_cache = requested_memory - 权重 - 激活峰值 - 非Torch内存

Scheduler 调度

  1. 一个 block 包含 block_size（默认16）个 token 的 KV，num_blocks = available_kv_cache // 单个 block 的字节数
  2. Scheduler 以 num_blocks 为容量上限，每条请求按 ceil(seq_len / block_size) 申请 block，当前调度所需 block 超过剩余 free block数时，暂停调度新请求或抢占低优先级请求，以此避免真实的显存 OOM
